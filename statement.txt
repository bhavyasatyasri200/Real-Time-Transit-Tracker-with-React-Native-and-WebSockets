Description
1. Project Overview and Background
Public transit applications like Citymapper, Transit, and Google Maps are cornerstones of urban mobility. Their core value proposition rests on providing accurate, real-time information. A delay of even a few seconds in reporting a bus's location can lead to a missed connection and a frustrated user. This project challenges you to build the foundational technology for such an app: a live map that displays vehicle positions updating in real time.

You will tackle this problem by creating two main components: a mobile application built with React Native and a backend service built with Fastify (a high-performance Node.js framework). The communication between them will be handled by WebSockets, a protocol designed for persistent, low-latency, bi-directional communication, making it ideal for real-time updates.

The backend will simulate a GTFS-RT (General Transit Feed Specification - Realtime) feed. GTFS-RT is the industry standard for disseminating real-time public transportation data. While you won't be parsing a real GTFS-RT binary feed, your backend's WebSocket broadcast will mimic the continuous stream of vehicle position updates that a real-world transit app would consume.

The frontend will not just display dots on a map. A key requirement is to create a seamless user experience by smoothly animating the vehicle markers between position updates. Instead of markers jumping or 'teleporting' every few seconds, you will use interpolation to create the illusion of continuous movement, a critical feature for a polished, professional application.

Why this stack?
React Native: Allows you to build a cross-platform mobile application for both iOS and Android from a single codebase, leveraging the power of React.
react-native-maps: The de-facto standard for integrating native maps (both Google Maps on Android and Apple Maps on iOS) into a React Native application. It provides the necessary components for rendering markers, polylines, and other map elements.
Fastify: A modern, low-overhead Node.js web framework. Its speed and simple plugin architecture make it perfect for building a high-performance WebSocket server to simulate our real-time feed.
WebSockets (ws library): Unlike traditional HTTP request-response cycles, WebSockets maintain a long-lived connection between the client and server. This allows the server to push data to the client instantly, which is essential for real-time applications and far more efficient than constant client-side polling.
2. Core Technologies Deep Dive
Before you start coding, it's essential to understand the key tools and concepts you'll be working with.

react-native-maps
This library provides a <MapView> component that acts as your canvas. You will use its child components to draw on the map:

<Marker>: Represents a single point on the map. You will use this to display vehicles. A crucial part of this task is making these markers animated. Instead of directly updating the coordinate prop and causing a re-render, you'll use an animated value that smoothly transitions between old and new coordinates.
<Polyline>: Represents a series of connected coordinates, forming a line. You will use this to draw the vehicle routes on the map.
To use react-native-maps, you'll need to follow the installation instructions carefully, as it involves native configuration for both iOS (including your Podfile) and Android (including API keys for Google Maps in AndroidManifest.xml). You can use OpenStreetMap tiles to avoid the need for Google Maps API keys if you prefer, which is a supported configuration.

WebSockets
Imagine you want to know a bus's location. With HTTP, you would have to ask the server every few seconds: "Where is the bus now? ... Where is the bus now? ..." This is called polling. It's inefficient, introduces latency, and scales poorly. WebSockets solve this by opening a persistent two-way communication channel. The client connects once, and the server can then push updates whenever new information is available. This is the 'push' model, and it's fundamental to real-time web and mobile applications.

You will use the ws library on your Fastify backend, which is a simple and powerful WebSocket implementation for Node.js. On the React Native client, you'll use the browser-standard WebSocket API, which is available globally in the React Native environment.

Fastify
Fastify is designed for speed. For this project, its key feature is its excellent support for plugins. You will use the fastify-websocket plugin to easily add a WebSocket server to your Fastify application. Your server will have two primary responsibilities:

A standard HTTP endpoint (/routes): To serve the static GeoJSON data that defines the paths vehicles will travel on.
A WebSocket endpoint (/ws): To accept client connections and broadcast simulated vehicle position updates.
GeoJSON
GeoJSON is a standard format for encoding geographic data structures using JSON. It's the lingua franca for web mapping. You'll use it to define your transit routes. A route will typically be a LineString feature, which is essentially an array of longitude/latitude coordinate pairs.

Example of a simple GeoJSON LineString:

{
  "type": "Feature",
  "properties": { "route_id": "A1" },
  "geometry": {
    "type": "LineString",
    "coordinates": [
      [-74.0060, 40.7128],
      [-74.0030, 40.7158],
      [-74.0000, 40.7188]
    ]
  }
}
You can create these files manually, or use an online tool like geojson.io to draw routes and export the data.

Animation with react-native-reanimated or Animated API
This is the most challenging and rewarding part of the frontend task. When your app receives a new coordinate for a vehicle, you cannot simply update the marker's state. Doing so would cause the marker to instantly jump from the old position to the new one. The user experience would be jarring.

Instead, you will use an animation library to interpolate the position. The process looks like this:

Store the vehicle's current position in an animated value object (e.g., useSharedValue from Reanimated or new Animated.ValueXY() from the Animated API).
When a new WebSocket message arrives with updated coordinates, you don't update the value directly. Instead, you trigger an animation (e.g., Animated.timing() or Reanimated's withTiming()).
You tell the animation function: "Over the next 3 seconds (or whatever your update interval is), smoothly transition the animated value from its current position to these new coordinates."
The <Marker> component's coordinate prop is bound to this animated value, so the library handles the frame-by-frame updates on the native UI thread, resulting in a buttery-smooth animation.
This approach ensures that animations run smoothly even if the JavaScript thread is busy, which is critical for performance in a React Native app.

3. System Architecture
The architecture for this system is straightforward but demonstrates a powerful pattern for real-time applications. The client (React Native App) establishes a persistent WebSocket connection to the backend (Fastify Server). The backend runs an independent simulation loop to generate data and broadcasts it to all connected clients.

Data Flow Explained:

Initial Load: When the app starts, it makes a one-time HTTP GET request to the /routes endpoint on the backend to fetch the GeoJSON polylines for all available transit routes.
WebSocket Connection: Concurrently, the app establishes a WebSocket connection to the /ws endpoint. This connection is intended to be long-lived.
Connection Maintained: The server accepts the connection, and it remains open. The server can now push data to the client at any time.
Backend Simulation: The Fastify server runs an internal timer (e.g., setInterval). Every 3 seconds, it calculates the new position for each simulated vehicle along its predefined route polyline. It then broadcasts a JSON payload with these updates to all connected WebSocket clients.
Client-Side Rendering: The React Native client listens for messages on the WebSocket. When a position update message is received, it triggers the animation logic for the corresponding vehicle marker on the map, smoothly moving it to its new location.

Step 2: WebSocket Connection with Exponential Backoff
Create a custom hook or service to manage the WebSocket connection. It's crucial to handle disconnects gracefully. If the connection drops, you shouldn't spam the server with reconnection attempts. Exponential backoff is a standard strategy where you wait progressively longer between each failed attempt.

// useWebSocket.js (simplified example)
import { useEffect, useRef, useState } from 'react';

const useWebSocket = (url) => {
  const [messages, setMessages] = useState([]);
  const ws = useRef(null);
  let reconnectAttempts = 0;

  const connect = () => {
    ws.current = new WebSocket(url);

    ws.current.onopen = () => {
      console.log('WebSocket Connected');
      reconnectAttempts = 0; // Reset on successful connection
    };

    ws.current.onmessage = (e) => {
      const message = JSON.parse(e.data);
      setMessages(message);
    };

    ws.current.onerror = (e) => {
      console.error('WebSocket Error:', e.message);
    };

    ws.current.onclose = () => {
      const delay = Math.min(30000, 1000 * Math.pow(2, reconnectAttempts));
      console.log(`WebSocket Disconnected. Reconnecting in ${delay}ms...`);
      setTimeout(connect, delay);
      reconnectAttempts++;
    };
  };

  useEffect(() => {
    connect();
    return () => ws.current.close();
  }, [url]);

  return messages;
};

Step 5: Implement the Bottom Sheet and ETA Calculation
When a marker is tapped, you'll need to open a bottom sheet with details. The ETA calculation will be a pure function you write.

ETA Calculation Logic:

Find the vehicle's current position and the coordinates of its next stop.
Calculate the straight-line distance (Haversine formula) or, for more accuracy, the distance along the polyline between the two points.
Use the formula time = distance / speed.
Convert the time to minutes and display it.
You must expose this logic on the window object for automated testing.

// In a utility file, and exposed on window in your main App.js for testing
// e.g., in App.js: useEffect(() => { window.calculateETA = calculateETA; }, []);
function calculateETA({ currentPosition, nextStopPosition, speedKmh }) {
    // Haversine distance formula implementation
    // ... returns distance in km
    const distanceKm = haversineDistance(currentPosition, nextStopPosition);
    if (speedKmh === 0) return Infinity;
    const timeHours = distanceKm / speedKmh;
    return timeHours * 60; // Return minutes
}

8. FAQ
Q: Why use WebSockets instead of just fetching the data every 3 seconds (polling)? A: Polling creates unnecessary network overhead and introduces latency. Each request has the overhead of establishing an HTTP connection. With WebSockets, the connection is established once, and the server can push updates with minimal overhead, resulting in faster, more efficient communication ideal for real-time needs.

Q: My marker animations are lagging or jerky. What can I do? A: This is a common performance issue in React Native. Ensure you are using the native driver for animations where possible (useNativeDriver: true). For map coordinates, this is not possible, so you must use useNativeDriver: false. The next step is to ensure your state updates are not causing excessive re-renders in your component tree. Use React.memo for your marker components and ensure the data you pass as props is memoized. Using react-native-reanimated v2 or v3 is generally more performant than the built-in Animated API as its animations run entirely on the UI thread.

Q: How do I calculate a vehicle's position along a polyline for the simulation? A: This is a classic geospatial problem. A simple approach is to treat the polyline segments as a sequence and track the vehicle's 'progress' as a percentage or distance. For a given progress, you find which two vertices of the polyline it's between and then linearly interpolate the coordinates. For production systems, a library like Turf.js is highly recommended, as it provides functions like @turf/along that do this precisely.

Q: Do I need to pay for Google Maps API keys? A: Not necessarily. react-native-maps can be configured to use OpenStreetMap (OSM) tiles, which are free. The setup is slightly different, but it's a great alternative to avoid billing and API key management for development projects. You can find instructions in the library's documentation.
